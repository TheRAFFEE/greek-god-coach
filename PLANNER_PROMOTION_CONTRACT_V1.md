# 2. Canonical source-of-truth definition

## Promotion rule

After promotion, the canonical daily training source of truth must be:

```ts
DailyTrainingSession
```

from `src/lib/training-planner.ts`, generated by a production-safe planner input adapter and, if required by the migration design, persisted or audit-hashed into `AppState`.

## Canonical invariants

1. **One date produces one canonical daily training session.**
   - Home, Train, Log, and recommendation copy must reference the same session ID.

2. **The planner owns daily session composition.**
   - `sessionType`
   - primary daily title/action
   - ordered blocks
   - lift/run/mobility/recovery/support prescriptions
   - combined load estimate
   - deload metadata
   - logging targets
   - completion target summary

3. **Readiness and progression apply once.**
   - The planner may consume readiness/progression results.
   - The app must not also separately modify the same workout via `adjustWorkoutForReadiness()` after planner output is generated.

4. **Deload remains metadata.**
   - `metadata.deload === true` may decorate `LiftDay`, `RunDay`, `LongRunDay`, `RecoveryDay`, etc.
   - No runtime path may create or require standalone `DeloadDay` for planner promotion.

5. **RecoveryDay remains distinct.**
   - Recovery sessions must remain `sessionType: "RecoveryDay"`.
   - Recovery must not collapse into `MobilityDay`.
   - Mobility may exist as support or a block, not as the primary type for recovery days.

6. **Support work must not change primary session type or logging modality.**
   - Core support may appear as `support: [{ kind: "Core" }]`.
   - Core support must not convert a run day into `LiftDay` or `HybridDay`.
   - Core support must not create run/lift logging targets unless a separate execution contract explicitly adds support logging.

7. **Logs are durable receipts; planner sessions are prescriptions.**
   - `RunLog`, `WorkoutSession`, `SetLog`, and summaries remain the durable record of what happened.
   - Planner outputs define what should be executed and where logs should attach.

8. **Legacy engines are domain signal owners until explicitly replaced.**
   - Readiness, progression, workout, running, nutrition, goal, and performance engines remain domain evaluators unless a future phase explicitly migrates ownership.

9. **Planner production fallback must be explicit and safe.**
   - Current adapter fallbacks are shadow-safe but not production-safe.
   - Promotion requires rejecting missing critical inputs or creating a clearly labeled safe unavailable/recovery session.

# 3. Ownership contracts by subsystem

## 3.1 Planner ownership boundaries

### Current Owner

Current owner is developer-only planner code:

- `src/lib/training-planner.ts`
  - `buildDailyTrainingSession()`
  - `DailyTrainingSession`
  - `DailyTrainingBlock`
  - `DailyWorkoutPrescription`
  - `DailyRunPrescription`
  - `DailyTrainingSupportItem`
- `src/lib/training-planner-adapter.ts`
  - `buildDailyTrainingSessionInputFromAppState()`
  - `buildPlannerSessionFromAppState()`

Current planner output is consumed only by developer panels in `src/app/page.tsx` when `plannerDebugEnabled` is active.

### Future Owner

The planner becomes the owner of daily training prescription and composition only:

- source plan resolution for a date
- `sessionType`
- deload metadata
- recovery-day preservation
- run/lift/mobility/recovery/conditioning/support block composition
- logging target references
- completion target semantics
- planner audit trail

The planner must not own:

- raw readiness calculation
- raw weekly progression calculation
- run progression engine internals
- workout set-by-set engine internals
- nutrition decisions
- persistence writes
- UI rendering
- direct mutation of logs or state

### Required Inputs

Promotion-grade planner input must include:

- `date`
- `currentWeek`
- source training plan or selected source workout collection
- current `AppState` identity fields needed for references
- `ReadinessEngineResult`
- `ProgressionEngineResult`
- `GoalTrackingEngineResult`
- running recommendation or explicit null with policy reason
- running engine result when required by progression/recommendation contract
- completed workout sessions for completion status
- completed run logs for completion status
- user preferences such as warmup/cooldown and available minutes
- production-safe source provenance

### Required Outputs

The canonical promoted output must include:

- stable `DailyTrainingSession.id`
- `date`, `currentWeek`, `dayIndex`
- `sourcePlan` with source identifiers and `resolvedSessionType`
- `sessionType`
- `metadata.deload`
- `status`
- `readinessStatus`
- `summary`
- ordered `blocks`
- `workout`, `run`, `mobility`, `recovery`, `support`
- `warmup`, `cooldown`
- `combinedLoad`
- `modifications`
- `warnings`
- `todayGoals`
- `auditTrail`
- production logging targets for all executable modalities

### Promotion Risks

- Current planner adapter can synthesize fallback readiness/progression/goal tracking results.
- Planner currently selects seed workouts directly and is not a persisted runtime source.
- Planner completion status is derived from legacy logs, not from planner session references.
- Planner output is not consumed by Home/Train/Log production paths.
- Planner does not yet define a complete execution contract for support work.

### Rollback Risks

- If the planner becomes runtime canonical without preserving legacy comparable outputs, rollback could change displayed workout/run titles and log references.
- If planner session IDs replace legacy workout IDs without a compatibility layer, existing logs may no longer map cleanly to Home/Train completion state.

### Required Migration Work

- Create production-safe planner input adapter.
- Add explicit adapter contracts for Home, Train, Log, Recommendations, and Persistence.
- Decide whether planner sessions are persisted fully, audit-hashed, or regenerated from versioned inputs.
- Add planner versioning and migration metadata.
- Keep legacy engine fallback during pilot.

## 3.2 Legacy engine ownership boundaries

### Current Owner

Legacy daily training composition is owned by several active paths:

- `src/app/page.tsx`
  - `getWorkoutForWeekDay()` selection
  - `adjustWorkoutForReadiness()`
  - `getTodayRunForDate()`
  - `generateRunningRecommendation()`
  - `generateDailyPrescription()`
  - automatic `adjustments` creation
- `src/lib/home-command-center.ts`
  - `buildHomeCommandCenter()`
- `src/lib/training-engine.ts`
  - `evaluateTraining()`
- `src/lib/coach-engine.ts`
  - daily prescription, readiness adjustment, running recommendation, set-by-set and post-workout recommendation helpers

### Future Owner

Legacy engines should be narrowed to domain services and compatibility/fallback only:

- `training-engine.ts`: compatibility snapshot during pilot; eventually deprecated as daily composer.
- `coach-engine.ts`: recommendation and coach-decision helpers only until recommendation ownership is reconciled.
- `home-command-center.ts`: Home model builder, not independent training source.
- `page.tsx`: orchestration and rendering only, not daily training derivation.

### Required Inputs

Legacy compatibility outputs must receive the same planner-derived or domain-derived inputs used by the planner so comparisons are meaningful:

- canonical date/current week
- readiness result
- progression result
- goal tracking result
- planner source workout reference
- planner run prescription reference

### Required Outputs

During pilot, legacy outputs should produce:

- comparable Home/Train fields
- legacy `TrainingEngineResult`
- warnings and recommendation copy for shadow comparison
- no canonical state mutations that conflict with planner

### Promotion Risks

- Legacy engines currently own parts of daily prescription, recommendation, readiness adjustment, and Home training display.
- `adjustWorkoutForReadiness()` currently modifies workouts before planner/debug paths consume derived state.
- `evaluateTraining()` composes warmup/workout/run/cooldown from scheduled workout/run but lacks planner session-type fidelity.

### Rollback Risks

- Removing legacy paths too early could break active Home, Train, and logging flows.
- Keeping legacy paths active too long can create duplicate prescriptions and duplicate automatic adjustment records.

### Required Migration Work

- Define a feature flag for planner runtime pilot.
- Preserve legacy engine outputs as fallback/comparison during pilot.
- Remove or neutralize duplicate daily training derivation only after parity gates pass.
- Add tests proving readiness/progression are not applied twice.

## 3.3 Home ownership model

### Current Owner

Current Home ownership is split between:

- `src/app/page.tsx`
  - derives `todayWorkout`, `adjustedTodayWorkout`, `todayRunDisplay`, `nextRunLabel`, `dailyPrescription`
- `src/lib/home-command-center.ts`
  - `buildHomeCommandCenter()` creates `HomeCommandCenterModel`
  - internally calls/evaluates readiness/progression/goal/training/performance if not provided
- `src/lib/training-engine.ts`
  - provides `trainingEngineResult` consumed by Home and Mission Control

### Future Owner

Future Home training display owner should be:

```text
DailyTrainingSession -> Planner Home Adapter -> HomeCommandCenterModel.training
```

`home-command-center.ts` may remain the owner of Home dashboard shape, but it must consume planner-derived training fields instead of independently selecting/composing today's workout/run.

### Required Inputs

Home adapter requires:

- canonical `DailyTrainingSession`
- `AppState` for current weight, nutrition, logs, and completion evidence
- `MacroTarget`
- readiness/progression/goal/performance engine results
- today date
- data-quality signals

### Required Outputs

Home must receive:

- planner-derived workout name or "No lift scheduled"
- planner-derived run name or "No run scheduled"
- planner-derived estimated duration
- planner-derived priorities/today goals
- planner-derived warnings
- planner-derived completion status
- session type and deload metadata where display requires it
- recovery-specific copy for `RecoveryDay`

### Promotion Risks

- Home currently displays `adjustedTodayWorkout.title` and `nextRunLabel`, not planner `summary.title` or planner run/workout names.
- Mission Control currently consumes `homeCommandCenter.trainingEngineResult`, not planner output.
- Home can recompute progression/readiness if options are missing, which risks non-identical planner and Home inputs.
- Duration-based run support exists in planner but Home still uses `getScheduledRunForTraining()`, which returns null for zero-distance runs.

### Rollback Risks

- A planner Home adapter can break dashboard copy if legacy fields are removed instead of bridged.
- If Home starts showing planner output while Train remains legacy, visible Home/Train disagreement becomes more severe.

### Required Migration Work

- Create `DailyTrainingSession -> HomeTraining` adapter.
- Decide whether `HomeCommandCenterModel.trainingEngineResult` remains during pilot as compatibility only.
- Make Home display and Mission Control consume a planner-compatible training model.
- Add parity tests for Home workout/run/duration/recovery/deload/support display.

## 3.4 Train ownership model

### Current Owner

Train execution is owned by `src/app/page.tsx`:

- `TrainScreen()`
- `TrainingPlan()`
- `ActiveWorkout()`
- `TrainRunLogger()`

Train inputs currently include:

- legacy adjusted `Workout`
- legacy `TrainingEngineResult`
- `scheduledRun`
- readiness
- running recommendation

Lift logging writes `WorkoutSession` and `SetLog` directly in `ActiveWorkout()`. Run logging writes through `saveTrainRunLog()`.

Planner train panels are developer-only:

- `src/lib/planner-train-preview.ts`
- `src/lib/planner-train-screen-v1.ts`

### Future Owner

Future Train execution owner should be:

```text
DailyTrainingSession -> Planner Train Execution Adapter -> Train execution model
```

Train UI owns rendering and user interactions. The planner owns executable session structure and logging targets.

### Required Inputs

Train adapter requires:

- canonical `DailyTrainingSession`
- `AppState`
- active/completed workout sessions
- completed run logs
- readiness status and warnings
- recommendation copy if still separate
- execution policy for support/core/conditioning blocks

### Required Outputs

Train execution model must include:

- ordered executable blocks
- lift execution block with source workout ID and exercises
- run execution block with planned distance and/or duration
- recovery/mobility block semantics
- support/core block display and optional logging policy
- logging target for each executable block
- completion status for each modality
- warnings and safety locks
- title/copy suitable for production UI

### Promotion Risks

- `TrainingPlan()` currently starts workouts from `displayedWorkout.id` and `displayedWorkout.exercises`.
- `ActiveWorkout()` writes logs directly using legacy workout/exercise IDs.
- `TrainRunLogger()` defaults duration from distance (`Math.round(plannedDistance * 11)`), which is unsafe for duration-only runs unless adapted.
- Planner support blocks are not executable in current Train.
- Current developer planner train screen is preview-only and must not be treated as production execution.

### Rollback Risks

- If Train starts writing planner session IDs without preserving source workout IDs, existing workout progression and historical summaries may break.
- If run logging starts from planner duration-only prescriptions without compatibility fields, old run logs may lose planned-distance expectations.

### Required Migration Work

- Create planner train execution adapter.
- Define lift/run/support/mobility/recovery block execution rules.
- Update logging defaults to accept distance and duration prescriptions.
- Preserve source workout/exercise IDs for workout engine compatibility.
- Add feature-flagged runtime pilot and legacy fallback.

## 3.5 Log ownership model

### Current Owner

Log ownership is distributed:

- `src/app/page.tsx`
  - Daily Check-In
  - Nutrition Logger
  - Body Metrics
  - Progress Photos
  - ActiveWorkout set logging
  - TrainRunLogger run logging
- `src/lib/run-logger.ts`
  - `saveTrainRunLog()`
  - `saveRunLoggerEntry()`
  - `buildRunLoggerRecord()`
- `src/lib/workout-logger.ts`
  - `saveWorkoutLoggerEntry()`
  - workout logger adapter functions, but active page lift logging mostly writes directly
- `src/lib/storage.ts`
  - persistence of `AppState`

Durable log types live in `src/lib/types.ts`:

- `RunLog`
- `WorkoutSession`
- `SetLog`
- `WorkoutSummary`
- `PostWorkoutRecommendation`
- `PlanAdjustment`

### Future Owner

Future Log ownership should be:

```text
Planner logging targets -> Log Adapter -> existing durable log records
```

The planner does not write logs. It defines where logs attach.

### Required Inputs

Log adapter requires:

- canonical `DailyTrainingSession`
- selected block/logging target
- user-entered execution data
- source workout/exercise IDs
- existing logs for upsert/replacement policy
- current user ID and date

### Required Outputs

Log adapter must emit existing durable records:

- `RunLog`
- `WorkoutSession`
- `SetLog`
- `WorkoutSummary`
- `PostWorkoutRecommendation`
- `PlanAdjustment`, if recommendation policy requires it

It should also emit planner reference metadata once persistence model is updated:

- `plannerSessionId`
- `plannerBlockId`
- `sourceWorkoutId`
- `sourceExerciseId`
- `plannerVersion` or audit hash

### Promotion Risks

- Current planner logging targets are metadata only and are not used by runtime loggers.
- `AppState` has no planner session references.
- Completion status in planner is inferred from legacy logs by date/workout ID.
- Support/core work has no durable logging contract.
- The direct set-logging path in `ActiveWorkout()` bypasses `workout-logger.ts` save helpers.

### Rollback Risks

- New planner references could be orphaned if rollback reverts UI but keeps stored logs.
- If logs are keyed only by planner session IDs, legacy completion and progression engines may not find them.

### Required Migration Work

- Extend log record schema or add a planner-log-reference model.
- Create planner-to-run-log and planner-to-workout-session adapters.
- Define support/core logging policy: display-only, checkbox, or structured log.
- Add backward-compatible completion queries that can resolve both planner and legacy IDs.
- Add backup/restore coverage before any persisted planner references ship.

## 3.6 Recommendation ownership model

### Current Owner

Recommendations are currently owned by `src/lib/coach-engine.ts` and active page effects:

- `generateDailyPrescription()`
- `generateRunningRecommendation()`
- `generateNextSetRecommendation()`
- `generatePostWorkoutAnalysis()`
- `createCoachDecisionLogEntry()`
- automatic adjustment creation in `src/app/page.tsx`

Running and workout engines also contribute domain signals:

- `src/lib/running-engine.ts`
- `src/lib/workout-engine.ts`

### Future Owner

Future ownership should be split:

- Planner owns final daily session prescription context.
- Recommendation engines own domain recommendation signals.
- A Recommendation Adapter owns final user-facing recommendation copy and `PlanAdjustment` side-effect policy.

The planner should not independently generate all coaching recommendations unless a later phase explicitly replaces coach-engine ownership.

### Required Inputs

Recommendation adapter requires:

- canonical `DailyTrainingSession`
- readiness result
- progression result
- running recommendation/domain result
- workout engine result where relevant
- macro/nutrition context
- prior post-workout recommendations
- existing adjustment logs for idempotency

### Required Outputs

Recommendation adapter must produce:

- daily recommendation copy aligned with planner session
- run recommendation copy aligned with planner run prescription
- set-by-set recommendations during lift execution
- post-workout recommendations after workout completion
- optional `PlanAdjustment` records with stable idempotency
- explicit conflict-resolution decisions

### Promotion Risks

- `generateDailyPrescription()` currently accepts a legacy workout, not a planner session.
- Running recommendation currently only runs when `plannedRunDistance > 0`; duration-only planner runs may receive no recommendation.
- Automatic adjustments in `page.tsx` may record legacy decisions that conflict with planner session output.
- Set-by-set and post-workout recommendation functions are not planner-aware.

### Rollback Risks

- If recommendation side effects are changed before planner promotion is stable, duplicate or conflicting `adjustments` may persist.
- Removing legacy daily prescription too early can break Home coach recommendation copy.

### Required Migration Work

- Define recommendation conflict policy:
  - planner session wins for daily modality/session shape
  - readiness/progression engines win for safety and weekly constraints
  - running/workout engines win for modality-specific progression signals
- Create `DailyTrainingSession -> DailyPrescription` compatibility adapter or replacement recommendation model.
- Make automatic adjustment creation planner-aware and idempotent.
- Add duration-run recommendation behavior.

## 3.7 Readiness ownership model

### Current Owner

Current readiness is owned by:

- `src/lib/readiness-engine.ts`
  - `evaluateReadiness()`
  - `readinessInputFromDailyCheckIn()`
  - `readinessInputFromWeeklyWindow()`
- `src/lib/coach-engine.ts`
  - `calculateReadiness()` compatibility wrapper

  - derives readiness from latest check-in and baseline
- `src/lib/home-command-center.ts`
  - may recompute readiness from state when not supplied
- `src/lib/training-planner-adapter.ts`
  - provides shadow-only fallback readiness when missing

### Future Owner

Future owner remains `readiness-engine.ts` for readiness calculation. The planner consumes readiness but does not own it.

Readiness contract:

```text
DailyCheckIn / weekly check-in window -> Readiness Engine -> ReadinessEngineResult -> Planner
```

### Required Inputs

Readiness requires:

- daily or weekly check-in fields
- sleep, soreness, stress, energy, alcohol, pain, pain severity
- resting HR and HRV if available
- baseline resting HR/HRV
- date/mode

### Required Outputs

Readiness must provide:

- `score`
- `status`
- `confidence`
- `reasons`
- `reason`
- `recommendation`
- `recommendationType`
- `trainingGuidance`
- `recoveryGuidance`
- `dataQualityWarnings`

### Promotion Risks

- Current app uses `calculateReadiness()` wrapper, not always the full `ReadinessEngineResult`.
- Planner debug paths reconstruct a simplified `plannerReadiness` with `confidence: "High"` and empty reasons/warnings.
- Home can recompute readiness if not supplied.
- Planner adapter fallback readiness is shadow-safe but not production-safe.
- `adjustWorkoutForReadiness()` currently applies readiness before planner/debug display, creating double-application risk.

### Rollback Risks

- If readiness results are persisted differently during promotion, rollback may display different readiness scores/copy.
- If planner-specific fallback readiness leaks to runtime, user may receive unsafe low-confidence prescriptions.

### Required Migration Work

- Pass full `ReadinessEngineResult` into planner in production.
- Remove simplified debug-only readiness from any production path.
- Prevent Home from recomputing readiness when a canonical readiness result exists.
- Disable legacy `adjustWorkoutForReadiness()` in planner-runtime mode.
- Add safety tests for Red/Yellow readiness applying once.

## 3.9 Persistence ownership model

### Current Owner

Current persistence owner is `AppState` in `src/lib/types.ts`, with local and remote persistence in:

- `src/lib/storage.ts`
  - `migrateAppState()`
  - `loadStateWithRecovery()`
  - `saveState()`
  - snapshots and recovery backups
- `src/lib/supabase-persistence.ts`
  - `syncAppStateToSupabase()`
  - table row mappers for workout sessions, set logs, meals, run logs, food scans, coach decision logs, macro target history
- `src/lib/backup-restore.ts`
  - backup payloads and restore flows, referenced by storage/page code

`AppState` currently has no planner session storage field.

### Future Owner

Persistence should own storage of planner references and reproducibility data, not planner computation itself.

Future persistence model must decide between:

1. Persist full `DailyTrainingSession` snapshots.
2. Persist planner session references plus source input hash.
3. Persist source workout ID + planner version + readiness/progression/goal result hashes.
4. Persist only log references initially and regenerate sessions while in pilot.

Recommended contract for pilot:

```text
Persist planner session audit reference + planner version + source IDs + input hashes.
Do not persist full mutable planner sessions until schema and migration are approved.
```

### Required Inputs

Persistence migration requires:

- planner session ID
- planner version
- date/current week/day index
- source workout ID
- resolved session type
- deload flag
- readiness/progression/goal tracking result references or hashes
- block logging targets
- legacy source IDs for rollback

### Required Outputs

Persistence must support:

- localStorage migration and normalization
- Supabase row mapping or deliberate local-only pilot flag
- backup/restore inclusion
- rollback compatibility
- deterministic reconstruction or clear audit explanation

### Promotion Risks

- Planner decisions are currently ephemeral and may change when source data or code changes.
- Supabase sync does not include planner sessions or planner references.
- Backup/restore does not account for planner session state.
- `AppState` has no planner field.

### Rollback Risks

- Persisting planner-only IDs without legacy source IDs could make rollback unable to read completion state.
- Supabase/localStorage divergence could occur if local has planner refs and remote does not.

### Required Migration Work

- Design `AppState` planner metadata field or separate persisted record type.
- Add `migrateAppState()` normalization.
- Add Supabase schema/table or explicitly keep planner pilot local-only.
- Add backup/restore schema inclusion.
- Add rollback migration plan before runtime pilot.

# 7. Critical blockers

1. **Planner is not runtime source of truth.**
   - `page.tsx` still derives daily workout/run/prescription through legacy paths.

2. **Home is not planner-backed.**
   - `buildHomeCommandCenter()` still composes `TrainingEngineResult` via `evaluateTraining()`.

3. **Train execution is not planner-backed.**
   - `TrainingPlan()` and `ActiveWorkout()` still use legacy `Workout` and `TrainingEngineResult`.

4. **Planner logging targets are metadata only.**
   - Runtime loggers do not consume planner `loggingTarget` or `executionTarget`.

5. **No planner persistence model exists.**
   - `AppState` has no planner session/audit reference.

6. **Readiness/progression can be applied more than once.**
   - `adjustWorkoutForReadiness()` currently modifies workouts before planner-aware paths.

7. **Recommendation ownership is unresolved.**
   - `generateDailyPrescription()` and automatic adjustments are not planner-aware.

8. **Production planner adapter is missing.**
   - Current adapter includes shadow-safe fallbacks that are unsafe for production runtime ownership.

---

# 8. Major blockers

1. **Duration-only run path is not promotion-safe across Train/Recommendation.**
   - Planner supports duration prescriptions; active run recommendation and run logger defaults remain distance-centric.

2. **Mission Control still depends on legacy `TrainingEngineResult`.**
   - `page.tsx` passes `homeCommandCenter.trainingEngineResult` into orchestration/modeling.

3. **Support/core execution and logging policy is incomplete.**
   - Planner extracts core support, but Train/Log do not own support execution.

4. **Hybrid/mixed modality execution requires hardening.**
   - Planner can represent mixed blocks, but production Train completion/logging is still legacy-modality based.

5. **Progression inputs are incomplete in Home-derived progression.**
   - `home-command-center.ts` passes null running/workout results into progression input.

6. **Direct set logging in `ActiveWorkout()` bypasses centralized workout logger helper.**
   - This complicates adding planner references consistently.

7. **Supabase sync would currently drop planner runtime metadata.**
   - No planner row mapping exists.

---

# 9. Minor blockers

1. **Planner readiness is reconstructed multiple times in `page.tsx` debug paths.**
   - This increases drift risk and currently sets confidence to `High` with empty warnings.

2. **Date/day-index logic is duplicated.**
   - `planDayIndex()` exists in planner and Home command center; page also computes day index.

3. **Current planner source provenance is seed-plan only.**
   - `sourcePlan.source` supports future values but runtime source integration is not implemented.

4. **Current Home/Train copy says Train is the only place to execute/log, but Log still owns other logging surfaces.**
   - This is a UX/source ownership clarity issue before broader promotion.

5. **Planner completion status is derived, not persisted.**
   - Acceptable for shadow mode, incomplete for promoted runtime ownership.

# 10. Recommended implementation sequence

## Phase 27C — Production planner input contract and adapter design

Documentation/test-design first.

- Define missing-input behavior.
- Define planner version and audit hash.

- Build `DailyTrainingSession -> Home training model` adapter.
- Do not remove legacy Home fields yet.
- Run Home parity tests against existing Home output.
- Preserve planner as non-promoted unless feature flag is explicitly enabled.

## Phase 27E — Train execution adapter

- Build `DailyTrainingSession -> Train execution model`.
- Support lift/run/recovery/mobility/support blocks.
- Add duration-run defaults.
- Preserve source workout/exercise IDs.

## Phase 27F — Log target and completion mapping

- Create planner logging-target adapters.
- Add planner reference metadata design.
- Prove completion status for lift/run/support without breaking legacy logs.

## Phase 27G — Recommendation reconciliation

- Create planner-aware daily prescription/recommendation adapter.
- Make automatic adjustments idempotent by planner session.
- Define conflict-resolution policy.

## Phase 27H — Readiness/progression sequencing cleanup

- Establish one canonical sequence:

```text
AppState/logs/check-in
  -> readiness/progression/domain engines
  -> planner
  -> Home/Train/Log/adapters
```

- Prove Red/Yellow/Recovery Focus apply once.

## Phase 27I — Persistence and rollback design/implementation

- Add planner metadata to AppState or sidecar state.
- Add localStorage migration.

## Phase 27J — Full shadow trial safety gate

## Phase 28A — Feature-flagged planner runtime pilot

- Readiness/progression sequencing is single-application

- shadow trial has zero critical blockers

# 11. Non-goals for planner promotion

The planner promotion contract does not require the planner to replace every domain engine.

Explicit non-goals unless a later phase says otherwise:

- replacing readiness calculation
- replacing weekly progression calculation
- replacing workout set-by-set recommendation internals
- replacing running progression internals
- replacing nutrition/macro engines
- replacing goal tracking
- replacing performance/physique engines
- mutating logs directly from planner
- changing source training plan data during planner session generation

---

# 12. Promotion contract summary

The planner can be promoted only when it becomes the single canonical daily training session provider while preserving domain engine ownership and durable log compatibility.

Required canonical flow:

```text
Domain signals + source plan + AppState
  -> production planner input adapter
  -> DailyTrainingSession
  -> Home / Train / Log / Recommendation / Persistence adapters
  -> UI and durable logs
```

Until that flow exists and passes the required safety gates, planner must remain:

- developer-only
- advisory-only
- read-only
- not promoted
