Phase 27A — Planner Promotion Readiness Audit

Audit mode: No code, tests, UI, source-of-truth, persistence, or implementation files were modified.

Planner status: developer-only, advisory-only, read-only, not promoted.

1. Promotion readiness score

38 / 100

Verdict

The planner is not ready to replace the legacy training engine yet.

It has made strong progress as a developer-only advisory prescription model, especially around RecoveryDay preservation, core support extraction, deload metadata, duration fidelity, primary session resolution, and preview/train-screen developer models.

However, it is still not safe as the app’s canonical runtime engine because the active app still depends on multiple legacy paths for Home display, Train execution, run/lift logging, readiness/recommendation/progression feedback loops, persistence, and source-of-truth state shape.

The planner currently consumes legacy engine outputs more than it owns them.

9. Critical blockers summary

  1. Planner is not the runtime source of truth. Home and Train still build from legacy paths.
  2. Train execution is still legacy-workout based. No executable production Train adapter exists.
  3. Logging targets are planner metadata only. Runtime logging still writes through legacy forms.
  4. Recommendation ownership is unresolved. Planner consumes recommendation outputs but does not replace them.
  5. Readiness/progression ownership is unresolved. Planner consumes these outputs but active app computes and applies them separately.
  6. Persistence has no planner state. No persisted planner session or planner audit artifact exists.

10. Major blockers summary

  1. Home model adapter missing.
  2. Planner Train UI is preview-only.
  3. Hybrid/mixed modality logging needs hardening.
  4. Duration-based run recommendation gap remains active-path risk.
  5. Source-plan ownership remains seed-based.
  6. Shadow/promotion audit tools are not sufficient as promotion gates.

11. Minor blockers summary

  1. Duplicate planner readiness construction in page.tsx.
  2. Regex classification still exists.
  3. UTC day-index assumptions should be validated before source-of-truth promotion.

13. Recommended promotion roadmap

Phase 27B — Planner Promotion Contract

Audit/design only. Define canonical ownership boundaries, planner input/output contracts, remaining engine ownership, planner non-mutation rules, migration requirements, and rollback requirements.

Deliverable: PLANNER_PROMOTION_CONTRACT_V1.md. No code changes.

Phase 27C — Home Adapter Plan + Implementation

Create a production-safe adapter: DailyTrainingSession -> HomeCommandCenterModel-compatible training fields.

Required proofs: Home workout from planner, Home run from planner, duration-based runs display correctly, RecoveryDay displays as recovery, deload metadata appears only as metadata, and no planner promotion unless behind explicit flag.

Phase 27D — Train Execution Adapter

Create adapter: DailyTrainingSession -> executable Train model.

Must support lift blocks, run blocks, recovery blocks, mobility blocks, conditioning blocks, support/core blocks, logging visibility, logging defaults, and hybrid behavior without duplicate legacy adjustment.

Phase 27E — Log + Persistence Mapping

Design and implement safe persistence model. Decide whether to persist full session, planner session ID only, planner audit hash, source workout ID plus planner version, or logging target references. Update localStorage migration, Supabase sync contract, backup/restore, and rollback plan.

Phase 27F — Recommendation Ownership Reconciliation

Define whether planner consumes, wraps, owns final daily copy, owns adjustment entries, or triggers run/workout recommendations. Eliminate conflicts between planner, daily prescription, running recommendation, post-workout recommendations, and automatic adjustments.

Phase 27G — Readiness/Progression Ownership Reconciliation

AppState/logs/check-in
  -> readiness/progression engines
  -> planner
  -> Home/Train/Log

Prevent double readiness adjustment, fallback readiness in production, and planner/legacy mismatch on recovery/deload days.

Phase 27H — Full Runtime Shadow Trial

Run planner and legacy side by side across all 84 seed days, readiness states, progression modes, deload metadata, duration runs, long runs, lift days, conditioning days, core support days, hybrid/mixed days, and logged/completed states.

Promotion gate should require zero critical logging mismatches, zero source-of-truth conflicts, zero persistence regressions, and documented expected differences only.

Phase 28A — Feature-flagged Planner Runtime Pilot

Use a hard feature flag: plannerRuntimeEnabled. Preserve legacy fallback, avoid data migration without backup, compare planner-vs-legacy runtime outputs, capture mismatches, and allow rollback.

Phase 28B — Promotion Decision

Promote only if Home uses planner, Train uses planner, Log uses planner logging targets, recommendations are reconciled, readiness/progression sequence is fixed, persistence/backup/rollback are proven, and shadow trial has no critical blockers.

14. Final recommendation

Do not promote planner yet.

The planner is strong enough to continue as a developer-only advisory/shadow system, but it is not yet safe to replace the legacy training engine.

Recommended next phase: Phase 27B — Planner Promotion Contract V1.

That phase should be audit/design-only and should define the exact promotion contract before any runtime integration changes.
